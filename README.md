# winbuddy

Разговорный ИИ-помощник для Windows. Ты пишешь ему обычным языком —
«проанализируй диск D», «наведи порядок в загрузках», «почисти мусор»,
«просканируй на вирусы» — а он раскладывает задачу на шаги и выполняет их
через встроенные средства Windows. Мозг — Claude (tool use), руки — набор
проверяемых Python-функций. Есть консольная версия и десктопное приложение
на Electron.

Главная идея проекта не в наборе функций, а в **безопасности**: ИИ, которому
доверили двигать и удалять файлы, опасен по определению, поэтому всё построено
вокруг того, как не дать ему ничего сломать.

---

## Что умеет

- **Анализ дисков** — что занимает место, крупнейшие файлы, состояние дисков и памяти.
- **Поиск мусора** — временные файлы и кэши с разделением по уровню риска.
- **Раскладка по папкам** — сортирует файлы по категориям (Изображения, Документы, Видео…).
- **Чистка мусора** — удаление только в корзину (обратимо), с делением на безопасный мусор и кэши приложений.
- **Проверка на вирусы** — оркестрация встроенного Windows Defender.
- **Разговорный интерфейс** с памятью контекста в рамках сессии.

---

## Модель безопасности

Это ядро проекта. Каждое потенциально опасное действие проходит несколько
независимых барьеров:

1. **Разделение read / write.** Анализ и отчёты агент делает свободно.
   Перемещение, удаление, переименование — только с подтверждением человеком.
2. **Подтверждение вне контроля модели.** Решение «выполнять или нет»
   принимает человек через интерфейс, а не модель. Агент лишь *предлагает*
   план; выполнение запускает код обвязки после явного «да». Даже если модель
   ошибётся в тексте — действие идёт по снимку плана, а не по её словам.
3. **dry-run по умолчанию.** Опасная операция сначала показывает план
   («перемещу 47 файлов вот так»), и только после подтверждения выполняется.
4. **Удаление только в корзину.** Прямого удаления в коде нет — всё через
   send2trash, любое действие откатывается.
5. **Allowlist / forbidden-list путей.** Системные папки (Windows,
   Program Files, корзина, System Volume Information на любом диске)
   заблокированы намертво. Запись возможна только внутри явно разрешённых
   корней. Пути нормализуются через ntpath со схлопыванием "..", обойти
   защиту трюком "..\..\windows" нельзя.
6. **Аудит-лог.** Каждое реальное write-действие пишется в logs/audit.log
   до выполнения.
7. **Деление мусора по риску.** Безопасный мусор (temp, .old, thumbs.db)
   удаляется по умолчанию; кэши живых приложений трогаются только по явному
   запросу.

## Архитектура

```
             CLI (rich)                Electron (окно, JS)
                  \                         /
                   \                       / websocket
              agent/loop.py          agent/server.py (FastAPI)
                   \                       /
                    \                     /
                  agent/core.py  (единое ядро агент-цикла)
                          |
                  tools/registry.py  (описания + диспетчер)
                    /      |       \
          analysis.py  actions.py  security_tools.py
           (read)      (write)      (Defender)
                          |
                     safety.py  (проверка путей — источник правды)
```

Одно ядро агента (core.py) обслуживает и консоль, и десктоп-приложение.
Различаются только способ вывода шагов и способ подтверждения — они вынесены
в callback'и (emit, confirm). Это позволило добавить GUI, не переписывая
логику агента.

**Стек:** Python (pathlib, shutil, subprocess, send2trash, psutil),
Claude API (агент-цикл на tool use), FastAPI + WebSocket, Electron.

---

## Как устроен агент-цикл

1. Пользователю задаётся цель, Claude получает её и список инструментов.
2. Claude выбирает инструмент, код выполняет его, результат возвращается в диалог.
3. Шаги повторяются, пока Claude не соберёт достаточно данных для ответа.
4. Для write-инструментов между «выбрал» и «выполнил» вставлен барьер
   подтверждения человеком.

## Запуск

### Требования
- Windows, Python 3.11+, Node.js (для Electron-версии)
- Ключ Claude API (официальный или через совместимый прокси)

### Установка
```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
copy .env.example .env
```

### Консольная версия
```powershell
python main.py                     # интерактивный агент
python main.py --test-tools "D:\"  # проверка инструментов без API (бесплатно)
python main.py --self-check        # проверка скелета и безопасности
```

### Desktop-версия (Electron)
```powershell
cd electron
npm install
npm start
```
Или двойной клик по winbuddy.bat в корне проекта (нужен Node в PATH).

## Конфигурация (.env)

```
ANTHROPIC_API_KEY=...            # ключ API
ANTHROPIC_BASE_URL=...           # адрес провайдера (для прокси; пусто = официальный)
WINBUDDY_MODEL=...               # имя модели
WINBUDDY_WRITE_ROOTS=D:\orgtest  # где разрешена запись (через запятую)
```

Запись по умолчанию запрещена везде — папки для write-операций добавляются
явно, это осознанный безопасный дефолт.

## История разработки

Проект собирался фазами, каждая проверялась на живой системе перед следующей:

- **Фаза 0** — скелет, конфиг, слой безопасности, self-check.
- **Фаза 1** — read-only агент с инструментами анализа (MVP).
- **Фаза 2** — первый write-инструмент (раскладка) + механизм подтверждения + память диалога.
- **Фаза 3** — чистка мусора в корзину с делением по риску.
- **Фаза 4** — интеграция с Windows Defender.
- **Фаза 5** — Electron-приложение (FastAPI + WebSocket, единое ядро).

Подход read-only-до-действий окупился: он вскрыл несколько реальных проблем
до того, как агент получил «руки» — например, что корзина и системные тома
есть на каждом диске (а не только на C), и что счётчик мусора нестабилен
между сканами.

## Известные ограничения

- Подсчёт размеров использует логический размер файла; для sparse-файлов,
  junction-точек и дедуплицированных данных он завышен. Реальный размер на
  диске стоит считать через Windows API.
- Память диалога живёт только в рамках запущенной сессии.
- Отмена (undo) выполненных действий пока не реализована, хотя аудит-лог
  для этого уже накапливает данные.
